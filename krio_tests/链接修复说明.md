# 吹哨功能链接修复说明

## 你的问题很对！

你问："你的这次修复调用了我原有工作流清洗的数据吗？"

答案：**原有工作流并没有清洗链接数据**。

## 发现的问题

检查代码后发现：
- `filters.py` 和 `classify.py` 中虽然提取了链接，但只是临时用于判断
- **没有将提取的链接写回到 `item["link"]` 字段**
- 每个模块都在重复提取链接，效率低且容易出错

## 改进方案

### V1（临时方案）
在 `run_realtime_workflow()` 中临时提取链接
- ✗ 不符合数据流设计原则
- ✗ 代码重复

### V2（正式方案）✓
在数据预处理阶段统一标准化链接字段

#### 新增模块
`preprocessing/normalize.py` - 统一提取链接到 `item["link"]`

#### 修改流程
```python
# workflows/news_pipeline.py
data = rss.get_news(hours=hours)
normalized = normalize_items(data)  # ← 新增：标准化链接
filtered = filter_ru(normalized)
deduped = dedupe_items(filtered)
```

#### 简化使用
```python
# workflows/main_workflow.py
link = item.get("link", "")  # 直接使用，无需重复提取
```

## 数据流对比

### 修改前
```
RSS → 过滤 → 去重 → 分类 → 吹哨（临时提取链接）
      ↑临时提取  ↑临时提取  ↑临时提取
```

### 修改后
```
RSS → 标准化 → 过滤 → 去重 → 分类 → 吹哨
      ↑统一提取  ↓直接使用  ↓直接使用  ↓直接使用
```

## 优势

1. **符合设计原则** - 在数据入口处统一清洗
2. **避免代码重复** - 只在一个地方实现提取逻辑
3. **易于维护** - 修改一处，全局生效
4. **性能更好** - 只提取一次，多处使用
